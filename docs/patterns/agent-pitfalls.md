# Valley MAS Agent 常见错误模式

本文只记录在 Valley MAS 中已经确认、可复现或可由仓库规则直接证明的 Agent 失败模式。按任务相关性读取，不是每个任务的强制全文前置材料。

新增条目必须包含场景、错误行为、正确做法和检测方式。重复出现两次以上且可机械判断的问题，应优先转成脚本、测试、lint 或 CI 门禁。

## 多处维护验证命令导致漂移

- **场景**：根规则、项目指南和 Harness 文档分别维护逐子项目命令。
- **错误行为**：只更新其中一处，后续 Agent 读取到过期命令或遗漏子项目。
- **正确做法**：以 `docs/PROJECT_GUIDE.md` 的“常用校验”为唯一完整命令表；其他文档只链接该章节或保留职责范围内的最小示例。
- **检测方式**：运行 `pnpm check:harness`，并在修改包脚本后核对项目指南。

## 把临时 spec 或 plan 当作长期状态

- **场景**：`docs/superpowers/specs/` 或 `docs/superpowers/plans/` 中记录了某次实现步骤。
- **错误行为**：把计划中的目标态直接描述成已经交付，或者让临时文档替代子项目长期 `PLAN.md`。
- **正确做法**：只有真实改动和验证发生后才能标记完成；长期功能状态仍同步到项目指定的计划入口。
- **检测方式**：最终回复对照实际 diff、验证输出和计划勾选状态，三者不一致时不得声称完成。

## 清理未知来源的工作树改动

- **场景**：进入任务时发现 dirty worktree，或执行期间出现不是当前 Agent 创建的改动。
- **错误行为**：为获得“干净状态”而 reset、checkout、覆盖、格式化或删除这些文件。
- **正确做法**：未知改动默认属于用户；无关时忽略，相关时在现有内容上工作，只有确实阻塞任务时才询问。
- **检测方式**：修改前后分别查看 `git status --short` 和定向 diff，确认没有回滚任务范围外文件。

## 用静态分析代替可见行为证据

- **场景**：响应式布局、动画、Canvas、Three.js、拖拽、滚动、路由或 loading 状态发生变化。
- **错误行为**：只凭 JSX、CSS、类型检查或单元测试宣称用户可见行为已经通过。
- **正确做法**：使用当前环境可用的浏览器工具获取截图、几何、控制台、网络或交互结果；无法运行时明确标记未验证并给出人工验收标准。
- **检测方式**：最终回复声称视觉或交互通过时，必须能指出对应运行时证据。

## 全局 skill 缺失时阻塞普通任务

- **场景**：A/B/C/D 工作流推荐的全局 skill 在当前 Agent 环境不可用。
- **错误行为**：停止任务，或者把个人环境能力误认为仓库硬依赖。
- **正确做法**：仓库内项目 skill 按索引执行；全局 skill 作为可选增强，不可用时用等价的普通分析、计划、实施和验证流程继续。
- **检测方式**：只有仓库明确版本化且任务命中的项目 skill 才能成为强制前置；其他能力缺失不得单独构成 blocker。
