# Valley MAS 架构决策指南

> 只在共享模块、跨应用复用、协议或 Provider 迁移、复杂状态协调等有真实架构取舍的任务中读取。它不要求为普通页面和局部修复新增抽象、包或 ADR。

## 1. 先判断是否真的需要架构动作

优先保持改动局部。满足下列任一信号时，再讨论模块边界或长期设计：

- 同一能力已有两个独立调用方，且它们需要稳定共享行为。
- 新旧协议、模型 Provider 或第三方服务需要并存、灰度或迁移。
- 同一交互规则散在多个页面或 effect 中，已出现行为漂移或重复修复。
- 改动会改变跨应用依赖方向、公开契约、持久化数据或不可逆迁移策略。

单一调用方、短期试验和仅有“以后也许会复用”的设想，默认留在当前 feature 或子项目中。第二个独立调用方出现前，不提前抽共享包。

## 2. 共享能力与宿主装配分开

共享模块只承载稳定的领域行为、纯 UI 或可复用规则；路由、页面壳、权限入口、宿主 store、埋点和页面级弹窗装配保留在调用方。

当多个调用方的数据源或操作不同，但展示和行为一致时，优先通过小而明确的 props、callback、slot 或数据 hook 注入差异，而不是复制一份组件。注入点必须有当前消费方；未使用的“万能 adapter”不新增。

共享模块的公开入口只导出已有调用方需要的稳定符号。内部实现、临时工具和宿主特化类型保持局部，调用方不得深层 import 模块内部文件。

## 3. 依赖方向以实际边界为准

- 应用或页面负责装配；feature 负责当前业务流程；共享包只提供跨子项目稳定复用能力。
- UI 组件不直接依赖传输协议或页面级请求细节；请求、缓存和响应归一化放在现有 API 封装、hook 或服务边界。
- 业务 UI 消费面向界面的模型，不直接扩散后端 DTO、数据库模型或上游 Provider 字段。字段转换和错误归一化集中在边界处。
- 跨 feature 复用先经过明确公共入口；若两个 feature 需要互相深层 import，优先把真正共享的部分上提或由上层装配传入。

不为全仓强行套用 L0/L1/L2 目录。只有某个共享边界已经稳定、并且违反方向会持续造成问题时，才补对应的静态检查或 review 规则。

## 4. 多协议与多 Provider：核心只认 canonical 模型

当同一能力要兼容多个 AI Provider、事件流协议或外部服务时：

1. 先定义业务核心真正需要的最小 canonical 模型和行为。
2. 核心状态机、领域规则与 UI 只消费该模型。
3. 每种传输或 Provider 差异留在 adapter 中完成转换、降级和错误归一化。
4. 新的长期主路径应尽量贴近 canonical；即将淘汰的旧路径承担翻译成本。

canonical 不应吸收某一上游的全部字段。只纳入至少两个调用方或核心行为都需要的概念；其余业务特化事件和字段留在调用方扩展层。

## 5. 复杂交互：决策纯化，副作用外置

对于多面板互斥、流程状态、响应式规则、重试或异步竞态等复杂协调：

- 先把“输入状态 → 可见状态 / 下一步动作”收敛为纯函数或 resolver，并用决策表写清优先级与不变量。
- React effect、订阅、定时器、浏览器测量、动画和本地存储只负责读取、写入或驱动结果，不承载业务决策。
- 同一 DOM 节点的测量或订阅只保留一个所有者；其他消费者复用其结果，避免重复 observer 和状态竞争。
- 单测优先覆盖 resolver 的决策表、边界状态和错误分支；动画、拖拽、Canvas 等仍按项目规则补运行时证据。

## 6. 记录真正的架构决策

只有出现长期取舍时，才在现有 `docs/specs/` 或对应计划中记录：背景、选定方案、明确不做的事、被否决方案及理由、迁移步骤、回滚边界和验证方式。

以下情况通常不需要新增设计文档：一次性组件抽取、局部命名调整、单调用方的内部重构，以及没有改变公开契约的实现替换。

## 7. 实施前快速检查

- 这是否已有两个独立调用方，还是假设性的未来复用？
- 调用方需要知道的接口、错误语义和约束是否足够小且稳定？
- 宿主装配、传输字段和领域规则是否被混在同一层？
- 是否可以把复杂条件写成可测试的纯 resolver？
- 新的 adapter 或共享模块是否有明确的当前消费者和删除路径？
- 是否需要在现有 spec 或 plan 中记录不可逆取舍？

如果上述问题多数答案是否定的，选择更局部、更直接的实现。
