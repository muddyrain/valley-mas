# Valley MAS 测试治理策略

> **状态**：生效中。本文是项目级测试目标、历史测试债治理和测试取舍的唯一真源；完整验证命令仍以 [项目指南](./PROJECT_GUIDE.md#常用校验) 为准。

## 1. 目标与边界

目标不是一次性追求数字覆盖率，而是让新改动不再持续积累无测试债，并按业务风险逐步为既有功能建立可靠回归保护。

- 每个新增功能、修复和会改变运行时行为的重构，都要评估并落下与风险相称的自动化测试。
- 单测优先保护可确定断言的业务规则、边界条件、错误分支和数据转换；不以快照数量或覆盖率数字代替行为验证。
- UI 的响应式、动画、Canvas、Three.js、拖拽、滚动和真实网络交互仍需要运行时证据；jsdom 单测不能单独证明这些行为正确。
- 纯类型、自动生成代码、无行为的 re-export、纯样式微调和一次性原型不强制新增单测，但交付时必须说明理由及采用的替代验证。

## 2. 新改动的最低要求

### 2.1 必须优先测试的范围

下列改动原则上必须新增或更新单测：

- 解析、校验、格式化、规则计算、状态转换、权限判断和错误映射。
- 前端 store、hook、请求封装，以及会改变数据读写或交互状态的组件。
- Go 的 service、handler、workflow、模型约束和数据库读写分支。
- 线上事故、用户反馈的回归缺陷和 AI 输出解析/降级分支。

测试应断言用户或调用方可观察到的输入、输出与副作用，而不是内部实现细节。外部网络、时间、随机数和昂贵的上游依赖应替换为可控的 fake 或 mock；被测模块本身不得被 mock。

### 2.2 UI 与运行时边界

组件测试适合覆盖渲染状态、可访问性、事件与状态切换。出现以下情况时，还必须取得相应的运行时证据：

| 场景 | 最低补充证据 |
| --- | --- |
| 响应式、布局、滚动、loading | 截图或关键元素几何/可滚动性结果 |
| 动画、拖拽、Canvas、Three.js | 真实交互后的截图、几何或控制台结果 |
| 请求、路由、鉴权流程 | 明确的网络请求、URL 或端到端交互结果 |

不要求每个 UI 改动都增加永久 E2E 测试；是否沉淀取决于核心路径、回归概率和维护成本。

### 2.3 例外与交付说明

无法新增单测时，最终交付必须说明：未测的具体范围、原因、已执行的替代验证，以及仍存风险。不得以“时间不足”“改动很小”作为可测试业务逻辑的例外理由。

## 3. 测试放置与实现原则

- 先遵循目标子项目的 `AGENTS.md` 与现有测试布局；不为统一风格而迁移已有测试。
- 新建纯逻辑单测优先与源码同目录同名（例如 `parser.ts` 对应 `parser.test.ts`）；已有 `tests/` 目录且局部规范要求使用时，延续该规范。
- 测试命名描述可观察行为与预期；每个测试应独立、确定、可重复运行。
- 先测试底层规则，再测试 hook/store，最后测试组件；避免通过大型 UI 测试间接覆盖本可独立验证的业务逻辑。
- 不为了指标而测试框架、第三方组件内部实现或无运行时行为的类型文件。

## 4. 历史无测试功能的治理

**不一次性补齐全仓历史功能。** 当前不同子项目的测试基建和风险差异很大，一次性补测会阻塞正常迭代，也容易产生只为凑数、维护价值低的测试。

采用“止血 + 风险清债”的双轨策略：

1. **止血（立即生效）**：未来新增功能、bugfix 和行为重构遵守第 2 节，不再新增无测试债。
2. **随改补测**：触及已有高风险逻辑时，先为本次会改变的行为补回归测试；不要求顺手覆盖整个历史模块。
3. **事故驱动补测**：每个确认的线上或用户回归缺陷，在修复时增加能复现该缺陷的测试；若只适合运行时验证，记录其可重复验收步骤。
4. **专题清债**：以独立计划处理高风险模块，不与无关功能迭代捆绑。优先级为：鉴权与权限 → 数据写入/迁移 → 支付或提醒等不可逆副作用 → AI 解析与降级 → 公共 API/共享包 → 核心用户路径。
5. **建立基线后再设阈值**：所有主要前端子项目都能稳定运行测试、前端 CI 已接入测试并生成可信报告后，再单独决定按包或按增量的覆盖率门槛；在此之前不设置全仓统一百分比。

## 5. 推进里程碑

| 阶段 | 完成条件 | 当前决策 |
| --- | --- | --- |
| A：规则生效 | 本文、根协作规则和文档索引就位 | 本次完成 |
| B：测试基线 | 盘点各子项目测试入口、可运行性和高风险无测试模块 | 后续独立任务 |
| C：前端门禁 | 为已具备测试基建的包建立统一测试入口并接入 CI | 基线完成后实施 |
| D：风险清债 | 以专题计划补齐 P0/P1 历史风险模块 | 持续进行 |
| E：增量覆盖率 | 根据稳定基线设置分包或增量阈值 | D 前不实施 |

每个专题清债计划必须写明目标模块、用户/数据风险、预期测试层级、可运行命令与完成条件。不要将“已写计划”标记为“已补齐测试”。

## 6. 验收与演进

- 当前改动的测试是否真正执行，由各任务的最终交付报告如实说明。
- CI 与本地验证的完整命令只维护在 [项目指南](./PROJECT_GUIDE.md#常用校验)。
- 重复出现且可机械判断的测试缺口，优先升级为测试、静态规则或 CI 门禁，再考虑追加协作提示。
- 覆盖率是发现盲区的信号，不是单独的质量结论；测试应始终围绕回归风险和可观察行为。
