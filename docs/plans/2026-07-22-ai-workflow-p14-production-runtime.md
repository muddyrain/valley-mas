# AI 工作流 P14.1：生产运行底座实施计划

> 状态：进行中。首个切片只交付可靠异步运行与定时触发；公开 API、Webhook、外部写入审批和团队协作分批进入后续 P14 子阶段。

## 目标

让 owner 私有的已发布工作流能够脱离浏览器请求可靠运行。用户可创建、查看、停用定时触发器；每次触发生成可追溯的运行，并在服务重启或多实例部署后继续由可用 worker 接管。

首个验收场景：用户为一个不含文件输入、且不含写入型 capability 的已发布工作流设置 Cron；到点后系统仅创建一次运行，事件和结果仍可在现有工作流编辑器中查看。

## 本切片边界

- 使用数据库持久化任务和租约领取，不新增 Redis、消息队列或调度第三方依赖。
- 工作流运行冻结在入队时的图快照与 AI App 版本，不读取后续草稿。
- 首期仅支持手动异步入队与 Cron 定时；文件输入工作流继续使用既有前台试运行路径。
- 写入、模型存储等副作用工作流不得由 Cron 自动触发；Webhook、公开 API、自动发布、外部连接器写入与审批节点不在本切片实现。
- 重试只覆盖尚未开始或可安全重试的无副作用任务；不得自动重放生成封面、创建博客草稿等动作。

## 设计决策

1. 新增 owner-scoped `workflow_run_jobs`，保存运行 ID、冻结图、版本、触发来源、幂等键、调度时间、租约、尝试次数和终态错误码。
2. 新增 `workflow_triggers`，首期只保存 `cron` 表达式、时区、启停状态、下次触发时间和最近触发时间；每个 trigger 在单个时间槽只能创建一个 job。
3. `WorkflowRun` 增加 `queued` 状态和触发元数据。现有 `running/success/error/cancelled` 数据保持可读，现有同步 SSE 入口暂不移除。
4. worker 由 server bootstrap 启动，以短轮询领取到期 job，并通过租约避免多实例重复执行；执行时复用现有 Graph v4 校验、运行事件、工具权限和安全预览逻辑。
5. 取消 queued job 直接进入 cancelled；取消 running job 写入持久化取消请求，worker 在节点边界读取该状态。跨进程取消不再依赖进程内 `sync.Map`。

## 验收标准

1. 同一个 trigger 和时间槽在多实例并发扫描下最多创建一条运行。
2. worker 重启或租约过期后，未完成 job 可安全恢复或标记为可诊断失败，不会静默丢失。
3. 已禁用 trigger 不会创建新运行；取消 queued job 不会执行任何节点。
4. 定时触发拒绝文件输入、写入型或模型存储 capability 的图，并指出具体节点。
5. 普通运行、历史、SSE 续接和 owner 隔离没有回归；真实浏览器可完成创建、停用和查看一次 Cron 运行。
