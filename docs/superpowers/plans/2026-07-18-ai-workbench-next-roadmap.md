# AI 工作台后续路线图：P12—P14

## 产品锚点

Valley 的目标不是复制 Coze 的全量节点目录，而是成为“内容创作与个人知识自动化”中最可靠的 AI 工作台：用户能用自然语言与画布共同构建流程，理解每次运行结果，并安全地复用自己的知识、提示词和工具。

后续阶段必须遵守以下顺序：先让一次创建—运行—修复闭环可信，再扩展可复用资源，最后才开放生产自动化和高风险通用能力。

## 阶段总览

| 阶段 | 状态 | 核心结果 | 明确不做 |
| --- | --- | --- | --- |
| P11 | 已完成 | Graph v4、受控工具、画布编辑器、节点智能对齐辅助线与 AI operations | 旧工作流迁移、任意 HTTP/代码/SQL |
| P12 | 已完成 | 可诊断运行、可取消协作、结构化输出与回归用例 | 新增循环、批处理、外部凭据、插件市场 |

| P14 | 未开始 | 触发器、部署治理、团队权限和受控生产集成 | 先于审计与配额开放高副作用能力 |

## P12：运行可靠性与 AI 协作闭环

P12 已完成，详见：

- [规格](./2026-07-18-workflow-reliability-p12-spec.md)
- [实施计划](./2026-07-18-workflow-reliability-p12-implementation.md)

既有工作流数据按当前决策保留，不以清理作为 P11 或 P12 的前置条件。用户已在实际页面验收画布智能对齐、新 Graph v4 工作流链路，以及封面节点的新语义：提案不新增 Start 输入、默认直接生成一次，绑定已有上游布尔变量后 false 跳过且零生成调用、true 仅生成一次。2026-07-19 已在真实 ARK 环境验证立即停止后进入“已取消”终态，P12 的外部验收闭环完成。

P12 完成后，用户应能在同一处完成：描述需求、接受可解释的提案、运行、查看节点级原因、选择重试或让 AI 基于失败生成新提案。一次失败不能再表现为无原因的长时间加载。

## P13：资源中心与受控能力扩展

P13 只在 P12 验收完成后立项，分为四个独立可交付项，禁止合并成一次大改动。

### P13.1 Prompt 资源

- 已完成：个人提示词库已收敛为名称、描述和正文的普通资源列表；可创建、编辑、归档，并可在大模型节点中搜索、预览和插入。2026-07-19 已完成用户页面验收。
- 提示词库仅保存名称、描述和正文，归档但不做草稿、发布或版本。
- LLM 节点从词库预览并插入正文到当前系统提示词；插入后不保持绑定，也不会被词库后续修改自动更新。
- AI 优化只产生正文修改提案；实际效果通过承载它的大模型节点试运行验证。

### P13.2 知识库 2.0

- 实施中：现有工作流版本已冻结知识库绑定与检索策略，运行结果保存安全引用；文档来源、处理状态与安全切片预览已完成，下一步单独交付检索试验台。
- 文档处理状态、来源、切片预览、更新时间和引用可追溯。
- 工作流绑定知识库版本和检索策略；运行详情显示安全引用而不是全文泄露。
- 增加检索试验台，再评估重排、混合检索和多模态摄取。

### P13.3 工具资源与扩展节点

- 将 Tool capability 扩展为版本化资源：输入/输出 Schema、超时、重试、幂等性、副作用、成本、权限和审计策略。
- 首批只考虑受控 OAuth 连接器和内部工具；每个连接器先完成授权、撤销、调用日志和 owner 隔离。Notion OAuth 连接已进入实施：凭据只在服务端 AES-GCM 加密保存，授权 state 单次且限时；“工具”页只管理连接状态，不开放任意第三方调用。
- 图能力优先顺序为：多路 Switch、结构化 LLM 输出、文本/JSON 转换、受限批处理。

### P13.4 意图识别节点

- 已完成：Graph v4 提供独立“意图识别”节点，用户仅配置命名意图、说明、示例与文本输入；运行时固定复用受控 ARK 文本模型，并输出 `intentId`、`intentName`、`confidence`。每个意图与必填“其他”出口均在保存和运行前校验；未知模型意图固定降级到“其他”，无效模型响应保留节点级可诊断错误。2026-07-19 已由用户完成真实 ARK 页面验收。
- 将“意图识别”作为一等业务逻辑节点，而不是要求用户用大模型 JSON 输出再堆叠条件节点。首期仍复用受控文本模型，不引入第二套模型供应商或用户自带 Key。
- 节点输入为文本 query；用户配置有限、命名的意图及其说明/示例。节点固定输出 `intentId`、`intentName`、`confidence`，并为每个已配置意图及必填“其他”提供可校验的分流出口。
- 工作流版本冻结意图集合和出口映射；保存、AI 提案与运行统一校验不存在的意图、缺失“其他”出口和错误类型连接。运行详情只展示命中结果、置信度和安全错误分类，不展示模型内部推理。
- 首期不支持任意分类器训练、在线学习、隐式新增意图或直接执行副作用；分类失败必须进入“其他”或返回可诊断节点错误。

P13 不开放任意 HTTP、任意代码、SQL 或用户自定义数据库写入。它们要求 Secret 管理、网络出站策略、沙箱、审计、配额与人工审批全部到位后才可评估。

## P14：生产自动化与治理

P14 的目标是把已验证的工作流用于稳定生产，而不是继续增加画布功能。

- 触发器：手动、定时、Webhook、站内事件；每种都需要幂等键、队列、失败重放和停用开关。
- 人工审批：写入、通知、发布等副作用在执行前必须提供可审阅的批准步骤。
- 运行治理：并发、配额、模型/工具成本、超时、重试、告警和保留期。
- 发布与集成：环境区分、版本提升、公共 API/嵌入式调用、可撤销部署。
- 协作：团队空间、角色权限、资源共享范围、审计日志和密钥所有权。

循环、批处理、任意 HTTP、代码节点与外部插件市场只能作为 P14 后的独立专题。它们不是“补一个节点”，而是运行治理和安全模型的扩张。

## 优先级判断规则

只有同时满足以下条件的能力才可进入当前迭代：

1. 明确服务内容创作或个人知识自动化的核心任务。
2. 输入、输出、副作用和 owner 权限可由服务端声明和校验。
3. 失败可以定位到用户可理解的原因，并可安全重试或修复。
4. 有对应的 fake 上游回归和真实环境验收路径。

不满足上述条件的 Coze 能力只能作为参考，不进入 Valley 的近期节点列表。
