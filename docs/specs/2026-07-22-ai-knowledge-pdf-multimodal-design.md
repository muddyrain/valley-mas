# AI Knowledge 多模态 PDF 摄取设计

## 状态

代码已完成；待线上服务器安装 Poppler 并完成真实视觉模型验收。

## 目标

将知识库 PDF 从仅提取原生文本，升级为可处理扫描页、嵌入图片与矢量表格的可检索资料，同时保持 owner 隔离、异步索引和既有文本 PDF 的兼容性。

## 处理链路

```text
上传 PDF（可选择视觉模型）
  -> 保存原文件内容并进入 pending_parse
  -> Poppler 按页渲染 PNG
  -> 视觉模型逐页输出 OCR、表格 Markdown 与图片说明
  -> 合并原生文本与视觉结果
  -> 以页码、来源类型、段落边界切分
  -> ARK embedding + pgvector 检索
```

未选择视觉模型时，保留既有原生文本抽取和固定长度分段路径。选择视觉模型后，原生文本仍会入库；视觉结果作为独立来源片段补充，避免扫描件或视觉表格丢失。

## 数据与接口

- 文档保存 `vision_model_id` 与原始 PDF 内容，后台任务可重试解析而无需浏览器再次上传。
- 片段保存 `page_number` 与 `source_type`；安全预览和检索引用可显示“第几页、文本或视觉解析”。
- 上传接口新增可选 multipart 字段 `visionModelId`。仅 PDF 使用此字段，必须引用 owner 可用且具备 `vision` 能力的模型目录条目。
- 文档状态新增 `pending_parse`、`parsing`。解析或渲染失败会保留稳定错误码，用户可重试。

## 运行时与边界

- 服务调用系统 `pdftocairo`，不经 shell 拼接用户输入；临时目录与文件始终在任务结束后清理。
- 单文件沿用 2MB 上限；富解析额外限制页数、单页图像大小、视觉输出长度和总分段数，避免 CPU、磁盘和模型成本失控。
- Poppler 缺失、渲染失败、视觉模型不可用与模型上游失败分别映射为可诊断状态；不会中断 HTTP 服务进程。
- 当前模型目录与 provider 密钥策略不变，不新增模型环境变量。视觉模型由上传用户显式选择。

## 验收

1. 普通文本 PDF 不选视觉模型时保持现有摄取行为。
2. 带视觉模型的扫描 PDF 与表格 PDF 进入异步解析，生成按页可追溯的片段。
3. 上传页可选择视觉模型、查看处理状态，并在片段预览中看到页码和来源。
4. 缺少 Poppler、模型无权限、渲染失败和视觉上游失败有明确状态且可重试。
5. 本地与线上部署文档说明 Poppler 安装、检查和升级顺序。
