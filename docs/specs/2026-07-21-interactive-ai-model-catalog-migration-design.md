# 交互式 AI 模型目录全量迁移设计

## 目标

所有由用户主动发起、当前仍依赖 `ARK_*`、`GEMINI_*`、`LIFE_TRACE_AI_*` 或 `MIND_ARENA_AI_*` 的 AI 调用，统一改为从 AI 模型目录选择能力匹配的已启用模型。调用请求必须携带 `modelId`；服务端以目录记录决定 Provider、实际模型、超时与用量审计。

本设计不新增管理员场景默认模型，也不允许服务端按环境变量静默挑选业务模型。

## 范围

### 纳入

- Web：资源标题/标签建议、博客阅读问答与导读、博客封面生成、Valley AI Chat、AI 工作台的提案、提示词建议、头像生成、调试/对话与上下文副驾驶。
- Life Trace：今日建议、每周回顾、生活助手、图片分析、Pantry/衣橱拍照识别、备注与字段补全、菜谱、穿搭、Inbox 整理、书影音补全、轻账本对话等用户主动触发能力。
- AI Mind Arena：创建辩论时选择文本模型，并将模型锁定在该场辩论的服务端请求上下文中。

### 不纳入

- 后台异步、定时或自动触发的 AI 任务。它们没有用户可提交的 `modelId`，不能以隐式默认模型替代。
- 视频生成：当前模型目录只有 `text`、`vision`、`image`、`embedding` 四类能力，未定义视频模型契约；保留现有独立渲染链路，待新增 `video` 能力后另行迁移。
- 向量检索/Embedding 基础设施。它不是用户在单次交互中选择的生成模型。

## 交互与接口契约

1. 每个主动触发入口展示能力匹配的模型选择控件；未选择时不提交生成请求。
2. Web 继续复用 `ModelPicker`；Life Trace 与 Mind Arena 各自使用适配其设计系统的受控选择控件，但都请求同一个 `GET /api/v1/ai/models?capability=...`。
3. API 请求新增必填 `modelId`。流式请求通过 JSON body 或显式 query 字段携带，不能依赖浏览器本地状态。
4. 服务端使用 `aimodel.FindEnabledModel(..., capability)` 校验归属能力和启用状态，使用 `aimodel.ProviderFromEnv` + OpenAI-compatible 客户端调用。
5. 每次成功或失败均写入 AI 用量审计，至少包含 feature、provider、model、token、latency、error。
6. 历史会话、辩论、运行记录需要保存本次实际模型，确保后续展示和重放不被目录改动影响。

## 迁移原则

- 不为兼容旧调用保留“未传 modelId 则回退 ARK/Gemini”的隐藏分支。
- 同一交互一次选择的模型必须贯穿该请求的全部模型步骤；多能力链路应在每一步明确要求相应能力模型，而非用文本模型代替视觉或生图模型。
- 不能以环境变量模型名冒充目录模型 ID；环境变量仅保留 Provider 密钥和 Base URL。
- 模型目录不支持的能力不强行纳入本次迁移。

## 验收

- 每个纳入入口能选择目录模型，未选模型无法调用。
- 服务端拒绝不存在、禁用或能力不匹配的 `modelId`。
- SiliconFlow 与 Amux 的真实调用均能返回预期结果，并可在审计中追溯。
- Provider 配置缺失返回 503，上游失败返回 502，前端显示可行动错误。
- 不存在任何用户主动调用继续依赖 `ARK_*`、`GEMINI_*`、`LIFE_TRACE_AI_*` 或 `MIND_ARENA_AI_*` 来决定模型。
