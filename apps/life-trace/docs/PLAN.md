# Life Trace 产品计划

本文件是 Life Trace 唯一计划入口，覆盖产品现状、路线图、家庭库存设计和下一阶段 AI 拍照分析入库计划。

维护规则：

- 只在本文件维护 Life Trace 当前计划，不再新增并行的 MVP、Roadmap、Task 或 Pantry 计划文档。
- 顶部“当前状态”和“下一步优先级”必须保持可信；已完成内容沉淀到“已完成能力”，不要继续伪装成待办。
- 家庭库存、AI 图片能力、PWA 提醒、智能体入口都在本文件统一维护。
- 计划发生变化时，先更新本文件，再进入实现。

## 产品定位

Life Trace 是一个 AI 主导的个人生活计划、踪迹记录和家庭库存管理 PWA。它把天气、提醒、日程、打卡、图片记录、家庭库存和 AI 建议组合成一个生活闭环：用户先计划生活，再完成计划，并沉淀为可回看的生活踪迹；用户也可以用库存能力记录家中物品、保质期、使用状态和共享家庭空间。

核心闭环：

```text
今日天气 -> 创建生活计划 -> 完成计划 -> 自动生成生活踪迹 -> 在踪迹流中回看
```

家庭库存闭环：

```text
拍照/手动记录商品 -> AI 识别和补全信息 -> 用户编辑确认 -> 入库 -> 到期提醒/使用记录 -> 生成生活踪迹
```

## 当前状态

当前 Life Trace 已从静态 MVP 进入“登录门禁 + 云端生活数据 + 服务端 AI + 家庭库存 + AI 拍照入库”的早期产品化阶段。

- 历史周报保存、展开、删除和从“下周行动”生成计划。
- 家庭空间模型、家庭 API、个人空间回填、共享家庭加入/退出/转移/解散能力。
- Pantry 库存列表、创建、编辑、状态更新、删除、筛选、分页、家庭空间切换和库存提醒基础能力。

## 技术基线

- 前端：React + Vite + TypeScript。
- UI：Tailwind CSS + shadcn/ui dark-first 风格。
- 图标：lucide-react。
- 状态：Zustand + localStorage 兜底持久化；核心数据通过 Go API 同步。
- PWA：manifest + service worker。
- 后端：复用 Valley MAS Go 服务，Life Trace 能力集中在 `server/internal/lifetrace`。
- AI：复用现有 ARK 配置和服务端 AI 调用链路，不在前端直连模型。
- 图片：复用现有上传接口和服务端图片分析能力。

依赖策略：

- 默认不新增第三方依赖。
- AI 拍照分析第一阶段复用浏览器原生 `MediaDevices.getUserMedia`、`canvas`、现有上传接口和现有 ARK 视觉模型。
- AI 拍照入库默认把上传后的真实照片作为库存图片，不自动把真实照片或裁剪图写入 AI 缩略图字段。
- 自动抠图第一阶段不引入专门抠图库；AI 缩略图只在用户主动生成或主动选择“用实拍图做封面”后写入。
- 如果未来需要高质量透明背景抠图，再单独评估第三方能力，例如云端抠图 API、ONNX/SAM 类模型或图像处理库；评估时必须先说明依赖、解决的问题、取舍和替代方案，并由用户确认。

## 家庭空间与 Pantry 设计

家庭空间已经不再作为独立文档维护，本节是当前唯一设计入口。

### 核心概念

`Household` 是统一的“空间”抽象：

- `personal`：个人空间，系统自动创建，不可解散。
- `shared`：共享家庭空间，用户主动创建，可邀请成员加入。

`HouseholdMember` 表示用户和家庭的成员关系：

- 一个用户对某个家庭有一条成员关系。
- 角色包括 `owner`、`admin`、`member`。
- 状态包括 `active`、`left`、`removed`。

`LifeTracePantryItem` 表示库存条目：

- 业务归属：`household_id`。
- 操作归属：`created_by`、`updated_by`。
- 兼容字段：迁移期保留 `user_id`，但库存归属判断以 `household_id` 为准。

### 业务规则

- 每个用户始终拥有一个个人空间。
- 个人空间不可解散。
- 用户加入共享家庭后，个人空间仍然存在。
- 库存永远属于某个 household，不直接属于某个 user。
- 用户加入家庭时，不自动迁移个人库存。
- 家庭解散后，家庭库存保留在原 household 上，进入只读归档态。
- 普通成员可以退出家庭。
- owner 如果要退出且仍有其他成员，需要先转移 owner。
- owner 是最后一位成员时，应走解散家庭流程。

### 已落地能力

- 通用 household model。
- personal household 自动创建和回填支持。
- household API：列表、详情、创建、成员、邀请、加入、退出、转移 owner、解散。
- Pantry 按 household 查询和权限校验。
- Pantry 列表响应包含当前 household 信息。
- 前端 Pantry 家庭空间切换和家庭管理入口。
- 共享家庭 Pantry 查询测试。

### 待打磨能力

- 家庭成员管理 UI 的细节体验。
- 邀请码复制、过期、撤销和错误态文案。
- 解散家庭后的只读归档 UI。
- 从个人库存移动或复制到共享家庭库存。
- Pantry 操作生成更清晰的踪迹记录。

## AI 拍照分析入库计划

### 目标

从 AI 页新增“拍照分析商品”入口，进入独立拍照分析页。用户可以在线拍照或选择图片，系统自动识别商品主体、提取商品信息，用户确认或编辑后将商品加入 Pantry 库存。

这不是现有“图片生活分析”的简单改名。现有图片生活分析面向计划和踪迹；商品拍照分析面向 Pantry 入库。

### 用户路径

1. 用户进入 AI 页。
2. 点击“拍照分析商品”。
3. 页面跳转到商品拍照分析页。
4. 用户允许摄像头权限，直接拍照；也可以从相册选择图片。
5. 前端展示拍照预览，并允许重新拍照。
6. 用户点击“开始分析”。
7. 前端上传图片到服务端。
8. 服务端调用视觉 AI，返回商品识别结果和库存字段草稿。
9. 前端展示识别结果和入库确认入口，编辑表单通过底部 Sheet 打开。
10. 用户调整商品名称、分类、数量、单位、保质期、购买日期、存放位置、提醒规则、所属 household。
11. 用户确认入库。
12. 前端调用 Pantry 创建接口。
13. 入库成功后展示成功反馈，并可跳转到 Pantry 详情或继续拍下一件。

### 页面结构

新增页面建议路径：

- 前端路由：`/ai/photo-item-analysis`
- 页面文件：`apps/life-trace/src/pages/PhotoItemAnalysisPage.tsx`
- API 封装：复用或扩展 `apps/life-trace/src/api/pantry.ts`、`apps/life-trace/src/api/upload.ts`
- 服务端 handler：新增或扩展 `server/internal/lifetrace/pantry_ai_handler.go`

页面分区：

- 顶部栏：返回 AI 页、标题“拍照分析商品”。
- AI 扫描舱：暗色扫描网格、状态胶囊、扫描线、取景角标和商品主体提示。
- 拍照区：实时摄像头预览、拍照按钮、切换到上传图片；按钮文案必须保持单行。
- 预览区：拍照结果、重新拍照。
- 分析状态：上传中、分析中、失败重试。
- 识别结果卡：商品名称、类型、品牌/规格置信度、AI 说明。
- 编辑表单：复用现有库存编辑的底部 Sheet 形态承载入库字段，避免移动端长表单挤压扫描区。
- 入库操作：确认入库、继续拍下一件、查看库存。

视觉要求：

- 首屏不做普通表单卡片感，默认使用“AI 商品扫描舱”作为主视觉。
- 扫描舱使用 Life Trace 现有 dark-first 主题、`life-ai` 和 `life-trace` 功能色，不新增独立色系。
- 移动端 390px 宽度下，`待确认`、`处理中`、`已入库` 状态胶囊不得换行。
- `打开摄像头`、`从相册选择`、`重拍`、`开始分析`、`确认入库` 等主按钮文案不得换行。
- 上传中、分析中、入库中必须在按钮内部展示明确 loading 图标，不能只靠文字偏移表达状态。
- 取景区在移动端使用更紧凑的商品包装比例，避免操作区被底部 Tab 或浏览器底栏挤压。
- AI 识别后的确认入库表单必须用底部 Sheet，字段布局参考库存编辑抽屉，并在 360px 宽度下自动降为单列。

### 商品识别字段

AI 返回草稿字段：

- `name`：商品名称。
- `category`：分类，例如 食材、饮料、零食、药品、日用品、护肤、清洁、其他。
- `brand`：品牌，可为空。
- `spec`：规格，例如 `500ml`、`2kg`、`12片`。
- `quantity`：数量，默认 `1`。
- `unit`：单位，例如 件、瓶、盒、袋、个、包。
- `storageLocation`：建议存放位置，例如 冰箱冷藏、冷冻、厨房、浴室、药箱、储物柜。
- `expiresAt`：保质期日期，可为空。
- `purchaseDate`：购买日期，默认今天或空。
- `tags`：标签，例如 早餐、冷藏、儿童、宠物、药品。
- `confidence`：整体识别置信度。
- `warnings`：识别不确定、保质期不可见、需要用户确认等提示。
- `cropBox`：建议裁剪框，保留服务端响应兼容，但前端默认不再用它生成 AI 缩略图。

用户可编辑字段：

- 商品名称。
- 分类。
- 数量和单位。
- 购买日期。
- 保质期。
- 存放位置。
- 所属家庭空间。
- 默认提醒规则。
- 备注。
- AI 缩略图：默认留空，用户可在库存编辑弹框中主动生成或手动使用实拍图做封面。

### 自动抠图与封面策略

第一阶段不承诺高质量透明背景抠图，也不自动把 AI 拍照入库的真实照片复制为 AI 缩略图：

- 入库时保存原图 URL 到真实图片字段。
- AI 缩略图字段默认为空。
- 列表卡片优先展示真实图片；如果未来用户主动生成 AI 缩略图，再展示缩略图兜底。
- 库存编辑弹框的 AI 缩略图区域只展示真实存在的 `thumbnailUrl`，不再用真实图片伪装成 AI 缩略图。

后续如果要做真正透明背景抠图，需要单独评估：

- 云端抠图 API：效果稳定，但增加外部服务依赖和成本。
- 浏览器端模型：隐私更好，但包体和性能压力大。
- 服务端图像处理模型：可控性高，但部署和资源成本高。

### AI 解析规则

提示词目标：

- 只输出 JSON。
- 不确定时明确标记低置信度，不要编造品牌、保质期或规格。
- 如果图片中有多个商品，优先识别主体最大的商品。
- 保质期不可见时返回空值，并提示用户确认。
- 如果识别为药品、保健品、婴幼儿食品等敏感品类，只做库存记录建议，不给医疗或安全结论。
- 如果无法识别商品，返回“未知商品”草稿，并保留图片让用户手动编辑。

### 服务端接口草案

新增商品拍照分析接口：

```text
POST /api/v1/life-trace/ai/pantry-photo-analysis
```

请求：

```json
{
  "imageUrl": "https://...",
  "householdId": "123",
  "hint": "可选的用户补充说明"
}
```

响应：

```json
{
  "name": "牛奶",
  "category": "饮料",
  "brand": "示例品牌",
  "spec": "250ml",
  "quantity": 1,
  "unit": "盒",
  "storageLocation": "冰箱冷藏",
  "expiresAt": "",
  "purchaseDate": "",
  "tags": ["冷藏", "早餐"],
  "confidence": 0.78,
  "warnings": ["保质期没有清晰出现在图片中，请手动确认"],
  "cropBox": { "x": 0.12, "y": 0.08, "width": 0.72, "height": 0.84 },
  "summary": "识别为一盒牛奶，建议冷藏保存。"
}
```

创建 Pantry 仍走现有创建接口，避免 AI 分析接口直接写库。这样用户必须确认后才入库。

### 前端状态设计

页面状态：

- `idle`：等待拍照或上传。
- `camera-ready`：摄像头可用。
- `captured`：已有拍照结果。
- `uploading`：上传图片。
- `analyzing`：服务端 AI 分析中。
- `reviewing`：用户编辑确认。
- `saving`：入库中。
- `done`：入库成功。
- `error`：权限、上传、分析或入库失败。

失败处理：

- 摄像头权限失败：提示改用图片上传。
- 上传失败：保留本地预览，允许重试。
- AI 分析失败：提供空白入库表单，允许手动录入。
- 入库失败：保留 AI 结果和用户编辑内容，允许重试。

### 与现有能力的关系

- 复用 `AppImageUploader` 的上传能力，但拍照页需要额外支持摄像头实时预览。
- 复用 Pantry 创建、家庭空间选择和提醒规则。
- 不复用现有 `ImageAnalysisDrawer` 的业务语义；它仍负责图片生活分析。
- AI 页只做入口和最近分析记录，不承载完整拍照流程。
- Pantry 页负责查看、筛选和后续编辑。

### 验收标准

功能验收：

- 用户能从 AI 页进入拍照分析商品页。
- 用户能直接拍照，也能从相册选择图片。
- 图片上传后，系统能返回商品识别草稿。
- 用户能编辑所有关键库存字段。
- 用户确认后，商品进入当前选择的 Pantry household。
- 用户从共享家庭空间确认入库后，返回 Pantry 列表时仍停留在该家庭空间。
- 用户可以把没有保质期的物品作为普通库存记录保存。
- AI 失败时，用户仍能使用手动表单入库。
- 入库后可回到 Pantry 列表看到新商品。

体验验收：

- 摄像头权限失败时不阻断流程。
- 分析中、上传中、保存中都有明确状态。
- AI 扫描舱首屏看起来像视觉识别工具，而不是普通表单卡片。
- 390px 移动宽度下，状态胶囊和主要按钮文案保持单行。
- 360px/390px 移动宽度下，确认入库 Sheet 字段不重叠，按钮 loading 不挤压文字。
- 低置信度结果不会装作确定，必须提示用户确认。
- 多商品图片不会自动创建多个库存条目，第一阶段只创建用户确认的一个商品。
- 移动端按钮和文字不溢出、不遮挡底部导航。

安全和边界验收：

- 前端不保存真实 AI 密钥。
- 服务端校验用户登录态。
- 共享家庭入库前校验用户是 active member。
- 药品、保健品等敏感品类不输出医疗建议。
- 未经用户确认不自动写库。

## PWA 能力

- 已提供 `manifest.webmanifest`。
- 已生成 192px / 512px 应用图标。
- 已注册 service worker，并缓存应用壳、manifest 和图标；manifest 与图标采用网络优先刷新，降低应用名称或图标变更后继续使用旧缓存的概率。
- service worker 导航请求遇到托管平台 404 时会回退到应用壳，避免用户在 `/today`、`/profile` 等子路由点击“立即更新”后看到 `404: NOT_FOUND`。
- Vercel 部署配置已补 SPA rewrite，直接打开或分享子路由时统一返回 `index.html`。
- 已在全局应用壳提供安装和新版本提示；支持一键安装的浏览器可直接安装，不支持的 iPhone/Safari 仍提示通过系统分享菜单添加到主屏幕。
- 我的页提供 PWA 控制台，展示安装状态、离线缓存状态、更新状态、分享能力，并提供安装、检查更新、立即更新和分享应用入口。
- 我的页底部展示当前应用版本，格式为 `v产品版本+构建标识`，例如 `v0.1.0+abc1234`。
- 生产构建会把当前 commit 短 SHA 注入 service worker，让每次提交后的部署都能被浏览器识别为可更新版本。
- 应用分享优先使用系统分享面板，失败时回退为复制应用链接。
- 本地系统通知桥接已实现，待手机端手动验收后再决定是否推进服务端 Web Push。
