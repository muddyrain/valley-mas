# Game Doc Sync Guard

当代码实现与游戏设计文档出现分歧时，AI 下次进入任务会读到错误的设计基线，导致按过期设计写新代码。本 skill 防止这种情况发生。

## 触发场景

以下任一情况发生时，必须触发本 skill：

- 修改了任何游戏系统的核心参数（数值、条件、触发逻辑）
- 新增或删除了游戏功能、职业、种族、神力工具、建筑类型
- 实现过程中发现文档设计不可行，做了实际调整
- 新增了原文档未覆盖的系统或机制
- 用户明确说"更新一下文档"或"这个设计改一下"

## 需要同步的文档清单

| 变更类型 | 必须同步的文档 |
|---|---|
| 玩法机制、系统参数、种族/职业设定 | `docs/GAME_DESIGN.md` |
| 新增/删除/调整开发任务 | `docs/TASK_BREAKDOWN.md` |
| UI 布局、组件、调色板、资产规格 | `docs/UI_DESIGN.md` |
| 技术栈、目录结构、红线规则 | `AGENTS.md` |

## 执行流程

### 第一步：定位变更影响范围

```
确认本次变更涉及哪些系统：
  □ 地图/地形系统
  □ Agent/小人/FSM 系统
  □ 文明/资源/升级系统
  □ 战争/战斗系统
  □ 外交系统
  □ 神力系统
  □ 时间/昼夜/季节系统
  □ UI/HUD
  □ 技术架构
```

### 第二步：对照文档找到需要修改的段落

- 用 `grep_search` 在 `docs/` 目录搜索相关关键词
- 确认文档中的描述是否与实际实现一致
- 找到所有不一致的段落，列出清单再统一修改

### 第三步：更新文档

- 使用 `replace_string_in_file` 精确修改，不要大段覆盖无关内容
- 数值类变更（如触发阈值、HP 数值）直接改数字
- 机制类变更（如删除某状态）加注释说明原因和版本（例：`// 初版暂不实现`）
- 新增内容插入到对应章节，保持文档结构不变

### 第四步：编码检查

```bash
python ../../../../.codex/skills/encoding-guard/scripts/check_mojibake.py \
  docs/GAME_DESIGN.md \
  docs/TASK_BREAKDOWN.md \
  docs/UI_DESIGN.md \
  AGENTS.md
```

### 第五步：回复中说明

在最终回复中简短说明：
- 改了哪个文档的哪个章节
- 改了什么内容（一句话）

## 常见错误模式

- 只改了代码，没有同步文档
- 在文档里加了新内容，但没有删除旧的矛盾内容
- 参数改了数值，但文档里的数值示例没有同步
- 删除了某个功能，但 TASK_BREAKDOWN 里对应的 Task 没有标注已取消

## 版本标注约定

文档中暂不实现的功能用注释标注，不要删除（保留设计意图）：

```
// 后续版本扩展：冷战状态（初版暂不实现）
// v0.2 计划：船只渡海
```

这样后续版本升级时可以直接找到并激活，不需要重新设计。
