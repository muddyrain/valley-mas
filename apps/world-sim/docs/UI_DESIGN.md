## 一、整体设计原则

1. **少即是多** — HUD 元素尽量少，让玩家专注于观察世界演化
2. **像素统一** — 所有 UI 元素使用相同像素密度，不混用矢量和像素
3. **半透明叠加** — 面板使用半透明深色背景，不完全遮挡地图
4. **颜色克制** — UI 主色调来自统一调色板，与游戏地图色调区分但不冲突

## 三、字体规范

- **像素字体**（全局）：[Press Start 2P](https://fonts.google.com/specimen/Press+Start+2P)（Google Fonts 免费）
- 正文大小：`8px`（对应像素字体 1x）
- 标题大小：`16px`（2x）
- 数值/数字：同正文，使用等宽特性
- **禁止**使用任何非像素字体（如 系统默认 sans-serif）

### 4.1 M0 调试 HUD 临时布局

M0/M1 早期原型允许保留调试信息，但必须避免挤占左上角：

- 顶部栏只显示项目名/阶段，不放长操作说明。
- 左上角状态面板由独立 `UIScene` 渲染，固定在屏幕坐标，不随 WorldScene 相机缩放/平移。
- 状态面板显示光标格子、镜头中心格子、镜头状态、人口、选中单位摘要、库存汇总和建造状态。
- 状态面板需要额外显示势力摘要行：势力名、人口、领土格数；这是验证势力基础是否真正接入的最小可视信号。
- M1 调试局面默认至少显示两条势力摘要（人类营地 / 兽人部落），每条都应有非 0 人口和起始领土格数。
- 光标格子和镜头中心格子需要显示地形与归属；建筑完成后，被划入领土的格子应显示对应势力名。
- 未选中单位时只显示“单位：未选中”，不常驻展示所有单位信息。
- 资源库存和建造状态格式尽量短，避免挤压面板宽度。
- 操作说明放到底部细栏，避免与状态面板叠加。
- 单位头顶标签只在选中时显示性别、当前状态和核心生存数值，不显示长句。
- 单位归属需要有常驻小型势力色标识，不能只依赖单位描边或状态填充色。
- 左键仅用于选中单位；直接下达移动命令仅保留调试开关，不作为正式玩法提示。
- M0/M1 在正式 sprite 接入前，建筑必须使用可辨认的代码绘制占位外形，不能只用同质色块；木屋至少区分屋顶、墙体、门，施工中至少区分地基和进度条。
- M1 默认木屋应生成在各起始势力领土附近，不能沿用远离营地的旧调试坐标。
- M0/M1 在正式资源 sprite 接入前，资源点必须使用可辨认的代码绘制占位图标，不能只用同质色块；木材表现为树/原木，石料表现为石块，铁矿表现为带橙红矿点的岩石，食物表现为浆果/麦穗。
- M1 领土系统接入后，领土叠加层使用势力色 24% 透明度覆盖 owned tile，只在领土变化时重绘，不能随 WorldScene 每帧全量重绘。
- 地图不默认渲染临时首都旗标；势力核心应由真实建筑、HUD 或后续详情面板表达，避免把调试标记误认为可交互玩法对象。

### 6.3 单位 Sprite（16×16px，4 方向 × 4 帧行走动画）

> 注意：单位尺寸从 8×8 调整为 **16×16px**，与瓦片同尺寸，保证在 128×128 地图上有足够的外观辨识度。

#### 外观生成策略：固定 Spritesheet（方案 A）

每个「种族 × 职业」对应一套固定 Sprite，靠**职业装备外形**区分职业，靠**名字和性别图标**区分个体。整个 M0~M4 阶段均使用此方案，不做运行时外观随机化，美工成本可控。

```
人类农民   → unit_human_farmer.png   （草帽 + 锄头）
人类战士   → unit_human_warrior.png  （头盔 + 铁剑）
人类建造者 → unit_human_builder.png  （安全帽 + 锤子）
兽人战士   → unit_orc_warrior.png    （骨甲 + 大斧）
精灵弓手   → unit_elf_archer.png     （叶冠 + 弓箭）
矮人矿工   → unit_dwarf_miner.png    （矿灯帽 + 十字镐）
国王（各族）→ unit_*_king.png        （王冠，比普通单位高 2px）
```

> **为什么不做外观随机？** 16×16px 的尺寸在 128×128 的世界地图上非常微小，外观差异很难被玩家看到。WorldBox 本体也是固定 Sprite，个体识别靠名字和头顶标记，而非外观随机。

## 十、开发阶段美工策略

| 阶段 | 美工策略 |
|---|---|
| **M0 技术验证** | 地形可先用纯色方块；单位、建筑和资源点允许用代码绘制的简易像素占位外形，保证可辨识，不用正式美术资产 |
| **M1 最小原型** | 接入 Kenney 免费资产，能区分地形和单位即可 |
| **M2 完整循环** | 整理统一 Spritesheet，补全 UI 图标 |
| **M3 内容填充** | 委托或购买差异化 4 种族 sprite（参考 §七 方案C）|
| **M4 Web 性能优化** | 全套美术整合为单一 Spritesheet，减少绘制调用 |

> **建议**：M0/M1 阶段不投入正式美工资产，但关键对象不能退化成难以辨认的同质色块；先用代码绘制占位外形验证玩法，玩法跑通后再投入美工成本，避免返工。
