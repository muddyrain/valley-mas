# WorldSim - 设计评审工作流

> 版本：v0.1 · 日期：2026-05-16  
> 这份文档定义研发评审角色与评审顺序。它们不是标准 runtime agent，不参与 `SimLoop`，也不代表已经接入模型、工具或自动编排。

## 1. 当前状态

当前项目已经有游戏内小人 `Agent/FSM` 设计，但这里的“评审角色”不是游戏内智能体，也不是标准 AI agent 规格。

从 v0.1 起，设计评审可以按本文件组织；如果未来要定义真正可运行的游戏内 AI agent，应先补充专门的 runtime agent 规格文档。AI coding 的 Harness Engineering 属于仓库级协作体系，见根目录 `docs/HARNESS_ENGINEERING.md`。

## 2. 角色

### 2.1 熟练玩家视角

关注点：

- 游戏是否有持续观察欲
- 神力是否有反馈和破局感
- 文明成长是否能产生故事
- 信息是否足够可读
- 操作是否太重或太受限

输出：

- 最吸引人的 3 个体验点
- 最无聊或最困惑的 3 个体验点
- 一个"我会继续玩"的理由
- 一个"我会流失"的理由

### 2.2 玩法设计评审视角

关注点：

- 规则是否自洽
- 数值是否有明显失衡
- 系统之间是否能互相组合
- 是否存在只有文案、没有机制承载的设计

输出：

- 规则冲突
- 缺失的反馈链
- 可延后的内容
- 必须前置的核心机制

### 2.3 架构师视角

关注点：

- 分层是否清楚
- 状态真源是否唯一
- tick、命令、事件是否能扩展
- Worker、存档、配置、评测是否被提前考虑

输出：

- 架构风险
- 模块边界建议
- 数据流问题
- 不应过早实现的复杂度

### 2.4 平衡与数据视角

关注点：

- 资源产出和消耗曲线
- 战争触发频率
- 种族差异是否可观察
- 性能指标是否可测

输出：

- 需要记录的指标
- 最小压测场景
- 可能导致单一最优解的规则
- 建议调参顺序

## 3. 使用时机

- 新增核心系统前
- 修改 tick 顺序前
- 修改实体 schema 前
- 新增种族、建筑、神力、世界阶段前
- M0/M1/M2 里程碑结束时

## 4. 评审顺序

1. 架构师视角先看边界和状态真源
2. 玩法设计评审视角看规则闭环
3. 熟练玩家视角看体验和可读性
4. 平衡与数据视角看指标和压测

## 5. 输出格式

每次评审建议使用同一格式：

```md
## 评审范围

## 结论

## 必须修

## 可以后置

## 风险

## 建议验证方式
```

## 6. 边界

- 这些评审视角不替代最终设计决策
- 这些评审视角不等同于可运行 AI agent
- 这些评审视角不直接修改代码
- 任何建议落地前都要同步相关合同文档
- 如果评审意见冲突，以 `ARCHITECTURE.md`、`SIMULATION_CONTRACT.md` 和当前可运行实现为准
