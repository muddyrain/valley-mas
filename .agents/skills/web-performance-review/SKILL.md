# Web 性能审查

## 定位

这是面向 diff 的性能审查流程，不替代功能 review、类型检查或真实性能基准。只报告有明确运行时路径和证据的性能问题；没有足够上下文时，降级为待验证风险，不把推测写成缺陷。

不要求固定数量的审查 Agent，不自动修改代码、创建 MR 或运行基准测试。

## 审查流程

1. 读取完整 diff、变更目的和受影响业务文件；忽略 lockfile、纯格式化和测试文件，但不忽略 `package.json` 的依赖变化。
2. 识别关键路径：启动、首屏、长列表、滚动、输入、动画、高频事件或网络请求。
3. 对疑似问题读取 diff 外的调用方、清理逻辑或数据规模，确认实际触发条件。
4. 合并同一根因造成的连锁现象，只报告最上游问题。
5. 输出 findings、尚待运行时验证项和未审查范围；没有问题时明确说明。

## 审查维度

- **渲染与状态**：不稳定引用是否真实导致 memo 子树、高频 effect 或长列表重复渲染；避免把低频内联函数当作问题。
- **资源生命周期**：事件、订阅、timer、observer、WebSocket 和请求是否随卸载或依赖变化清理；无界数组、Map、Set 是否有淘汰策略。
- **主线程与布局**：是否在渲染、高频回调或循环中同步排序、大 JSON 解析、反复读写布局，或用高成本属性驱动高频动画。
- **网络**：无依赖请求是否形成瀑布，是否重复请求或扩大无关缓存失效范围。
- **包体积**：新增依赖、整包导入、首屏大模块是否进入关键路径；根据实际导入方式和构建配置判断。

## 证据与严重度

- `confirmed`：代码已证明触发路径和影响链路，可作为 finding。
- `needs_runtime_verification`：数据规模、调用频率或浏览器行为未能从代码确认；写明最小验证方法。

优先级使用项目现有 `P0` 至 `P3`：只有明确造成崩溃、无界增长或关键路径显著退化时才使用高优先级。低频微优化、风格偏好和没有测量依据的建议不作为 finding。

## 输出格式

```markdown
## Web 性能审查

- 范围与关键路径：...
- Findings：
  - P?｜类别｜文件:行｜触发条件｜证据｜最小修复建议
- 待运行时验证：条件、建议的浏览器或构建证据
- 未发现的问题或未覆盖范围：...
```

## 验证边界

真实的加载、帧率、内存、布局抖动和交互延迟结论，仍需使用当前环境可用的浏览器、构建产物或性能工具取得证据。浏览器不可用时，报告必须明确这是静态审查，不能宣称性能已验证。
