# 文档事实审计

## 定位

把代码、包脚本、路由、配置示例和类型定义作为事实来源，检查 Markdown 中可验证的断言是否已经过时。目标是消除会误导开发的文档，而不是重写文风或扩大全仓文档工程。

默认先审计、后修改：用户只要求检查时只输出报告；用户明确要求修复或更新时，才修改已确认过时的文档。

## 边界

- 不要求工作区干净，不 stash、不覆盖未知改动；只触及用户指定或本次已解析的文档范围。
- 不自动创建分支、提交、推送、MR、Issue 或外部文档。
- 历史归档、变更记录和设计回顾默认只读；除非用户明确要求，或其中存在会误导当前入口的失效链接。
- 不把无法证实的描述改成猜测。拿不准时保留原文并记为待人工确认。

## 流程

1. **确定范围**：优先使用用户指定文件；未指定时，以根 README、`docs/`、子项目 README 和局部 `AGENTS.md` 为候选，不扫依赖或生成目录。
2. **提取断言**：定位文档中的路径、命令、环境变量、接口、函数签名、路由、依赖版本、数据字段和行为描述。
3. **建立证据锚点**：对每条断言读取当前代码、脚本、配置或类型定义，记录“文档位置 → 当前事实位置”。行为断言必须读实现，不能只做字符串替换。
4. **分级**：
   - `confirmed_stale`：事实已明确变化或目标已不存在，可修改。
   - `still_valid`：能与事实对应，不动。
   - `uncertain`：无法由仓库事实确认，不动并在报告中说明。
5. **最小修复**：仅改 `confirmed_stale` 的句子、路径或章节；保留原有结构和历史语气。整个模块已删除时，优先删除或明确标记其已废弃，不能用编造的新内容替代。
6. **核验与回报**：逐项复查引用路径和命令；说明已改、未改和待确认内容。修改 skill、`AGENTS.md`、Harness 文档或索引时运行 `pnpm check:harness`。

## 审计报告格式

```markdown
## 文档事实审计

- 范围：...
- 已确认过时：文档:行 → 当前事实文件:行 → 建议或实际修复
- 仍有效：...
- 待人工确认：文档:行 → 缺少的证据
- 验证：...
```

## 禁止事项

- 不因目录名、字段名或命令名相似就判定过时。
- 不顺手改写未在审计范围内的文档。
- 不把“可能更好”的架构、产品建议伪装成文档事实更新。
- 不以自动提交或外部协作动作作为审计完成条件。
