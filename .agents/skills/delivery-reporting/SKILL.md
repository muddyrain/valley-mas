# Delivery Reporting

## 目标

让实现过程和收尾汇报保持可追踪，避免“代码改了但计划没同步”“计划写了但实际没落地”“验证没跑但说完成”等协作误差。

## 必须触发

- 用户要求“详细汇报”“告诉我改了什么”“下一步做什么”“每次更新要同步计划”等。
- 任务跨多个模块、多个阶段，且需要阶段性汇报。
- 改动会影响计划文档、长期文档、功能状态、接口路径、依赖策略、数据模型、产品方向或验收标准。
- 当前改动可能改变 `docs/`、子项目 `docs/`、README 或计划文档中的状态、路线图、验收标准。

以下情况通常不触发：

- 小型单点功能、局部 bugfix 或普通 UI 微调，且不改变计划、接口、数据模型或验收标准。
- 只因为改动跨了多个相邻文件，但不需要阶段性汇报。

## 工作规则

1. 不把计划当完成；只有真实文件改动、测试或验证发生后，才汇报为已完成。
2. 关键实现检查点说明：本次完成、影响范围、已验证、计划文档同步、下一步。
3. 每次代码或配置改动后，都要判断是否属于既有计划、是否需要更新计划文档。
4. 如果无需更新计划文档，也要在最终回复说明原因，例如“仅修复拼写，不影响计划”。
5. 如果需要更新计划文档，必须先更新再最终汇报。
6. 如果计划文档缺失、过期或互相冲突，先指出并收敛到当前项目指定的唯一计划入口。

## 检查点格式

推荐使用短小结构：

```markdown
本次完成
- ...

影响范围
- ...

已验证
- ...

计划文档同步
- 已同步：...
- 或无需同步：...

下一步
- ...
```

## 计划同步判断

需要更新计划文档的场景：

- 功能状态从待办变成已完成，或从已完成回退为待验证。
- 新增或删除功能入口、页面、接口、数据字段、依赖或验收标准。
- 用户明确提出新的产品方向、体验要求、技术约束或依赖策略。
- 发现计划文档里的路径、命令、接口、状态与当前代码不一致。

通常无需更新计划文档的场景：

- 纯格式化、拼写、局部样式微调，且不改变产品方向或验收标准。
- 临时调试、一次性截图、没有进入仓库的实验。

## 收尾要求

最终回复必须包含：

- 改了什么。
- 验证了什么。
- 计划文档是否同步；如果未同步，说明为什么。
- 仍需用户手动验收或后续推进的内容。
