# UI 文案边界护栏

这个 skill 用来约束“用户看到的文案应该是什么”，避免把 AI 或开发者的分析过程直接渗进界面。

## 什么时候必须触发

- 修改用户可见的标题、副标题、说明文案、按钮文案、Badge、空状态、引导语。
- 新增设置页、设置卡片、功能入口、功能总览、状态说明。
- 你发现自己想在页面里解释“这个设置有什么用”“这里为什么这样设计”“哪个页面会用到它”。

## 核心原则

用户界面里的文案，只应该服务用户当前任务、当前状态和当前决策。

不要把这些内容直接写进界面：

- 开发者自己的分析结论
- 页面结构说明
- 功能依赖关系解释
- 实现路径解释
- 使用教学式说明
- “这里 / 这个页面 / 这个入口 / 已经整理到哪里”这类元叙述

## 禁止写进界面的文案类型

- “这里会影响 Today 页面和 AI 建议”
- “这里只保留摘要和入口”
- “已经整理到提醒设置里”
- “这个设置用于……”
- “你可以在这里……”
- “切换后会直接跟随……”
- “新增或删除后会自动同步……并直接用这份清单”

这些话可以出现在：

- PR / 最终回复
- 注释
- 文档
- AGENTS / skill

但不应该默认出现在用户界面上。

## 允许的文案方向

- 当前状态：已开启、已关闭、09:00、法定工作日、4 个节点
- 任务导向：提醒设置、库存与家庭、每日打卡、简报时间
- 结果导向：立即预览简报通知、绑定推送、检查更新
- 必要且短的业务说明：新建商品默认继承这里，单个商品可单独改

## 改写方法

把“解释系统”改成“表达状态和动作”：

- 先写用户要做什么
- 再写当前是什么状态
- 如果必须补一句说明，只补业务上必要的一句

例子：

- 差：`天气城市会影响 Today 天气和每日简报`
- 好：`天气与简报定位`

- 差：`这里只保留摘要和入口`
- 好：`通知、节奏和库存提醒`

- 差：`切换库存或调整周末策略时不用再在首页里来回找`
- 好：`统一管理通知与提醒节奏`

## 自检清单

- 这句文案是在帮用户完成任务，还是在解释设计？
- 去掉“这里 / 这个页面 / 用于 / 会影响 / 已经整理”后，界面是否更干净？
- 这句文案如果放进最终回复会更合适吗？如果是，就不要放进 UI。
- 是否能用更短的状态词、标签词、动作词替代说明句？

## 校验

- 实际修改 CJK/非 ASCII 文案时，运行定向 `encoding-guard`。
- 如果仓库里有页面源码快照类测试，补一条断言，防止分析型文案回流。
