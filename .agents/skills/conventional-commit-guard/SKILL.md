# 智能 Emoji Conventional Commit

当任务涉及编写或执行 `git commit` 时，使用这个技能。

## 目标

分析代码变更，并创建合适的 commit。

## 执行流程

### 1. 检查变更状态

先检查 `git status`。如果没有变更，就退出。

### 2. 根据文件状态决定分析范围

1. **如果同时存在【已暂存】和【未暂存】的变更**：
   - **询问用户**：是否要将未暂存的变更一起提交？
   - 如果用户同意，将所有变更暂存 (`git add .`)，然后分析**所有**变更。
   - 如果用户不同意，则只分析【已暂存】的变更。

2. **如果只存在【未暂存】的变更**：
   - 将所有变更暂存 (`git add .`)，然后分析**所有**变更。

3. **如果只存在【已暂存】的变更**：
   - 只分析【已暂存】的变更。

### 3. 分析并提交

完成分析后，参考"拆分 Commit 的准则"自主决定是否拆分 commit。如果需要拆分，则直接创建任务计划进行拆分提交；否则，进行单次提交。所有提交都应符合 emoji conventional commit 规范。

## Commit 最佳实践

- **提交前验证**：确保代码已 lint、构建正确且文档已更新
- **原子提交**：每个 commit 应包含服务于单一目的的相关更改
- **拆分大型更改**：如果更改涉及多个关注点，请将其拆分为单独的 commit
- **Conventional commit 格式**：使用格式 `<emoji> <type>: <description>`

## 长度约束

1. 首行格式：`<emoji> <type>(scope): summary` 或 `<emoji> <type>: summary`
2. `summary` 默认使用中文，短句、明确、可读
3. 首行控制在 100 个字符以内
4. 默认只写首行，不自动附加正文或 trailers，除非用户明确要求

## 拆分 Commit 的准则

在分析 diff 时，请考虑根据以下标准拆分 commit：

1. **不同的关注点**：对代码库不相关部分的更改
2. **不同类型的更改**：混合功能、修复、重构等
3. **文件模式**：对不同类型文件的更改（例如，源代码与文档）
4. **逻辑分组**：如果分开更容易理解或审查的更改
5. **大小**：如果分解开来会更清晰的非常大的更改

## 强制执行规则（Agent 必须遵守）

1. 在生成 commit message 前，先查看最近提交风格（至少 5 条），避免风格漂移。
2. 首行必须是 `<emoji> <type>: <summary>` 或 `<emoji> <type>(scope): <summary>`。
3. `summary` 默认使用中文，短句、明确、可读，不写空泛词。
4. 默认提交信息**只写一行首行**，不自动附加长正文、解释段落或 trailers。
5. 只有在用户明确要求时，才允许写正文与 trailers。
6. 生成 message 后，先自检再提交：格式、语义、长度是否符合要求。

## 失败处理

- 若提交被 hook 拒绝，不允许只说"已失败"；必须立刻修正 message 并重提。
- 若用户指出风格不一致，必须优先对齐仓库历史风格并 `amend`。
- 若用户指出"太长"，必须改成短提交首行并优先 `amend` 修正历史。
