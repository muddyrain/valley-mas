# 任务完成度护栏
当任务同时包含分析、实施、验证、收尾时，使用这个技能。

## 目标

避免出现这些协作误差：

- 只做了任务拆解，却让用户误以为已经开始执行
- 只改了一部分，却让用户误以为整项任务已经完成
- 只说“会继续做”，但没有落到真实改动或验证上
- 中途回退、失败或被打断后，没有明确说明当前真实状态

## 什么时候必须触发

- 用户说“继续吧”“帮我更新吧”“直接改吧”
- 任务需要先列 task list，再逐项实施
- 任务跨多个文件、多个页面或多个链路
- 任务天然包含“改代码 + 跑校验 + 汇报结果”
- 你怀疑自己容易停在分析、计划或口头承诺，而不是实际执行

## 工作规则

1. 开始执行前，先区分当前阶段：分析、实施、验证、收尾，不能混着说。
2. 只有真的做了具体动作，才可以说“已经开始处理”。
3. 只有真正完成了用户请求中的核心改动，才可以说“已经处理好了”。
4. 如果目前只完成了 task list、思路、方案或部分探索，要明确说“已完成规划”或“已完成部分实施”，不能模糊成“已经在做完了”。
5. 如果任务被中断、回退或恢复到稳定版本，要明确说明现在的真实状态，再继续推进。
6. 最终回复前，至少做一次自检：用户最初要的东西，是否真的被改了、是否真的被验证了、是否还有未完成项。

## 自检清单

在每个重要阶段，至少快速过一遍：

- 我现在是在分析，还是已经进入实施？
- 我刚才说的进展，是否有对应的真实动作或真实产物？
- 用户要的是“规划”，还是“真正落地”？
- 当前是否还有未做完的子任务？
- 我有没有把“下一步要做”说成了“已经做了”？
- 我有没有把“部分完成”说成了“全部完成”？

## 进度表述规则

优先用这些更真实的说法：

- “我先把任务拆一下，然后开始落第一项。”
- “目前已经完成 A，B 还没开始。”
- “这一步还停留在分析，没有真正改代码。”
- “刚才那次修改我已经回退到稳定状态，现在重新开始落地。”
- “这次先完成了规划，还没有执行后续实现。”

避免这些容易误导的说法：

- “已经开始做了。”
  条件：实际上只列了计划或只看了文件。
- “已经更新好了。”
  条件：实际上只做了一部分，或者还没跑校验。
- “我继续做第一个任务。”
  条件：实际上还没有真正进入第一项的实施。

## 收尾要求

最终说明里要明确回答这 3 件事：

1. 这次用户要的任务，哪些已经真正完成。
2. 哪些只完成了部分，或者还没开始。
3. 这次有没有做验证；如果没做，要明确说没做。

## 校验

- 改完这个技能后，运行 [$encoding-guard](../encoding-guard/SKILL.md) 检查中文文案。
- 如果这次任务本身就是新增或修改 skill，再补跑 `quick_validate.py`。
