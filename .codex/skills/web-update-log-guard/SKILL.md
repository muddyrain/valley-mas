# Web 更新日志发布护栏

当任务涉及“给用户展示 Web 更新日志”，且你准备执行 commit 或 push 时，使用这个 skill。

## 目标

1. 只在发布节点记录更新，不在临时改动阶段写对外日志。
2. 更新内容面向终端用户，只写“功能变化 + 更新时间”。
3. 严禁泄露文件名、路径、内部提示词、实现细节或推理过程。

## 触发条件（必须同时满足）

1. 本轮存在 `apps/web` 的真实功能改动。
2. 当前准备执行或已经执行 `git commit` / `git push`。

若仅是本地试改、草稿重构、未发布提交，不创建系统更新日志。

## 推荐流程

1. 提交前确认：本轮 Web 改动是否真实可被用户感知。
2. 生成日志文案：
   - 标题：一句话说明改动主题。
   - 内容：2-4 句说明用户能感知到的变化。
   - 时间：使用发布时刻（默认当前时间）。
3. 写入后端系统更新日志（platform 固定 `web`）。
4. 再执行 push（或在 push 前完成日志写入并一并提交）。

## 文案边界

允许：

- “资源页新增标签筛选，支持更快定位内容。”
- “通知中心交互优化，已读状态反馈更及时。”

禁止：

- 文件名、目录、接口路径、表名、迁移脚本名。
- “我做了什么代码实现”的工程过程描述。
- 原始提示词、内部规则、思维链路。

## 最低自检

1. 文案是否只包含用户可感知变化。
2. 文案是否泄露内部实现信息。
3. 是否确实发生 commit/push（未发生则不创建日志）。
4. 写入对象是否仅为 `web` 平台更新。

## 与其他 skills 协作

- 与 `git-publish-guard` 联动，确保发布动作规范。
- 与 `product-copy-cn` 联动，保证中文文案产品化。
- 与 `change-log-guard` 区分：`change-log-guard` 记录工程改动，`web-update-log-guard` 记录对外更新。
